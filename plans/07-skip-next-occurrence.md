# Plan 07 — Skip a single occurrence

**Outcome:** From a recurring reminder's detail screen the user can skip just
the next occurrence ("skip tomorrow's 8am only") with one tap — the reminder
keeps its RRULE and stays enabled, that one instant vanishes from the list, the
hero, and the OS queue, and every later occurrence is untouched. An Undo
snackbar restores it.

## Prereq

None. Adds **no** new dependency — skips are stored in one new nullable column
and filtered with `dart:convert` (already available). Requires a schema bump
(3 → 4) and a `build_runner` regen of `database.g.dart`.

## Why not EXDATE (the format question)

CLAUDE.md mandates standard iCalendar RRULE via the `rrule` package and forbids
inventing a custom recurrence format. EXDATE is the iCalendar-standard way to
exclude single instances — but the installed `rrule: ^0.2.18` **does not support
it**: `RecurrenceRule` models only the RRULE property (verified — its
`fromString`/`getInstances` API has no EXDATE/RDATE parameter, and the package
has no exclusion concept). So we store the excluded instants ourselves.

This does **not** violate the never-store-derived-data rule: a skip is *user
input* (an explicit choice), not something computed from `startDateTime` + RRULE
— the same category as `snoozedUntil`, which is already a stored column. We are
not inventing a recurrence format; the RRULE stays standard and untouched. We
store an exclusion set alongside it, which is exactly what EXDATE is
semantically. // ponytail: home-grown skip set only because the pinned rrule
can't do EXDATE. Ceiling: not portable to a real .ics export. Upgrade path if we
ever add calendar export — emit these as EXDATE lines then.

## Design decisions

- **What we store:** a JSON array of `millisecondsSinceEpoch` ints, one per
  skipped occurrence, in a new nullable `skippedDates` text column. Null = no
  skips (existing rows migrate cleanly).
- **What the int is:** the exact `DateTime` value `nextOccurrences` emits for
  that occurrence (local wall-time, *before* the `offsetMinutes` lead-time
  subtraction). `nextOccurrences` builds each occurrence deterministically as
  `DateTime(d.year, d.month, d.day, d.hour, d.minute)` from the RRULE expansion,
  so the same occurrence recomputed on any later reconcile produces the
  identical millisecond value — exact-equality matching is safe, no fuzzy
  windows.
- **One filter, all callers correct.** The skip filter lives inside
  `nextOccurrences`. `effectiveNextFire`, the detail "next 5" list, the home
  hero, and `Scheduler._reconcile` all route through `nextOccurrences`, so a
  single `.where(...)` there drops the skipped instant everywhere — including
  the OS queue on the next reconcile — with no change to the scheduler, the
  hero, or `effectiveNextFire`.
- **Recurring only.** The Skip action is offered only when
  `rruleString != null`. Skipping a one-time reminder is just deleting it, which
  already exists — so the one-time branch of `nextOccurrences` is left alone.
  // ponytail: no skip on one-time reminders; delete covers that case.
- **Undo, but no skip-management screen.** A skipped occurrence otherwise has no
  UI to bring it back (it recurs next cycle, but that specific instance is gone
  until then). A snackbar Undo covers the mis-tap — the one realistic mistake —
  cheaply. // ponytail: snackbar Undo is the only un-skip affordance; a "manage
  skipped dates" list is YAGNI until someone asks.

## Steps

1. **Schema.** In `database.dart` add
   `TextColumn get skippedDates => text().nullable()();`, bump
   `schemaVersion` to `4`, and in `migration` add
   `if (from < 4) await m.addColumn(reminders, reminders.skippedDates);`.
   Regenerate: `dart run build_runner build --delete-conflicting-outputs`.

2. **Pure helpers + filter — `recurrence.dart`.** Add `import 'dart:convert';`
   and:
   ```dart
   /// User-chosen occurrence instants to exclude (epoch millis of the
   /// wall-time value nextOccurrences emits). Not derived data — explicit
   /// user input, like snoozedUntil.
   Set<int> decodeSkips(String? raw) =>
       raw == null ? {} : {for (final v in jsonDecode(raw) as List) v as int};

   String encodeSkips(Set<int> skips) => jsonEncode(skips.toList());
   ```
   Then in `nextOccurrences`, inside the RRULE branch only, filter **before**
   `take(count)` so a skip never shrinks the returned window:
   ```dart
   final skips = decodeSkips(r.skippedDates);
   return rule
       .getInstances(start: startUtc, after: afterUtc)
       .map((d) => DateTime(d.year, d.month, d.day, d.hour, d.minute))
       .where((d) => !skips.contains(d.millisecondsSinceEpoch))
       .take(count)
       .toList();
   ```
   (The `.map` moves ahead of `.take`; ordering is otherwise unchanged.)

3. **Repository — `skipNext` / `unskip`.**
   ```dart
   /// Skips the next RRULE occurrence (not a pending snooze). Returns the
   /// skipped occurrence, or null if nothing to skip / not recurring.
   Future<DateTime?> skipNext(String id) async {
     final r = await _db.getById(id);
     if (r == null || r.rruleString == null) return null;
     final next = nextOccurrences(r, DateTime.now(), 1);
     if (next.isEmpty) return null;
     final skips = decodeSkips(r.skippedDates)
       ..add(next.first.millisecondsSinceEpoch);
     await _db.upsert(r.copyWith(
         skippedDates: Value(encodeSkips(skips)),
         updatedAt: DateTime.now()));
     await _scheduler.reconcile();
     return next.first;
   }

   Future<void> unskip(String id, DateTime occurrence) async {
     final r = await _db.getById(id);
     if (r == null) return;
     final skips = decodeSkips(r.skippedDates)
       ..remove(occurrence.millisecondsSinceEpoch);
     await _db.upsert(r.copyWith(
         skippedDates: Value(encodeSkips(skips)),
         updatedAt: DateTime.now()));
     await _scheduler.reconcile();
   }
   ```
   Skip targets the next *occurrence*, deliberately ignoring any pending
   snooze — a snooze is a separate transient "remind me in N min" and is
   cleared by firing/passing. // ponytail: skip vs snooze kept independent; no
   combined "skip the thing that fires next" semantics.

4. **Detail UI.** Under the hero `TkHero` block (after line ~136), when
   `r.rruleString != null && nextFire != null`, add a tappable "Skip next"
   row/button styled like the existing header actions (reuse `TkIconButton` /
   the `Icons.skip_next` or `Icons.event_busy` icon + a `body(...)` label — no
   new widget). On tap:
   ```dart
   final skipped = await ref.read(reminderRepositoryProvider).skipNext(reminderId);
   if (skipped != null && context.mounted) {
     ScaffoldMessenger.of(context).showSnackBar(SnackBar(
       content: Text('Skipped ${DateFormat('EEE, MMM d').format(skipped)}'),
       action: SnackBarAction(
         label: 'Undo',
         onPressed: () =>
             ref.read(reminderRepositoryProvider).unskip(reminderId, skipped),
       ),
     ));
   }
   ```
   The detail screen already `ref.watch(reminderByIdProvider)` and rebuilds on
   any reminder change, so the "next 5" list and hero refresh automatically
   after skip/undo — no manual refresh.

## Done when

- `test/skip_test.dart` (new) passes:
  - **Filter (pure):** a `FREQ=DAILY` reminder whose `skippedDates` encodes the
    2nd upcoming occurrence's millis → `nextOccurrences(r, after, 3)` omits that
    instant and still returns 3 items (proves the filter runs before `take`),
    and every returned item is a real daily occurrence.
  - **Codec round trip:** `decodeSkips(encodeSkips({a, b}))` equals `{a, b}`;
    `decodeSkips(null)` is empty.
  - **Later occurrences untouched:** skipping the next occurrence leaves the
    following ones identical to the unskipped expansion (shifted by one).
  - **Repository + reconcile (DB + fake scheduler, reusing the
    `scheduler_test` fakes):** on a recurring reminder, `skipNext` returns the
    next occurrence, persists it in `skippedDates`, and a subsequent
    `reconcile()` schedules the occurrence *after* it — never the skipped
    fireAt. `skipNext` on a one-time reminder returns null and writes nothing.
    `unskip` with that same instant restores it (next reconcile schedules it
    again).
- `flutter analyze` clean; all existing suites (`recurrence`, `scheduler`,
  `detail_snooze`, …) still pass. The `database_default_test` still holds after
  the schema bump (defaults unchanged; `skippedDates` defaults to null).
- [ ] Manual (device/emulator): create a daily reminder; open detail; tap
  "Skip next" → snackbar shows, the first "next 5" entry advances by one day,
  the hero updates, and the skipped fireAt no longer rings (verify via the
  next-fire label / no notification at that time). Tap Undo within the snackbar
  → the occurrence reappears.

## Reconcile implications

None new. `Scheduler._reconcile` already rebuilds the OS queue from
`nextOccurrences`, which now filters skips — so the skipped occurrence is
dropped from the queue on the reconcile that `skipNext` triggers, with zero
scheduler-code change. Skips are stored (a real column), survive reboot/reinstall
like any other reminder field, and are recomputed against the live RRULE every
reconcile (idempotent, self-healing).

## Out of scope

- **Skip from the home screen (long-press).** The detail-screen action delivers
  the feature; a home long-press menu is a separate UI item — add if users ask.
- A "manage skipped dates" screen or the ability to skip an arbitrary future
  occurrence (only the *next* one is skippable here).
- Skipping one-time reminders (delete covers that).
- Exporting skips as real iCalendar EXDATE lines — relevant only if calendar
  export is ever added (it's out of scope for v1 per CLAUDE.md).
- Auto-pruning past skipped instants from the column (they're inert once the
  occurrence is behind `after`; the list is tiny). // ponytail: prune only if a
  reminder somehow accumulates thousands of skips.
