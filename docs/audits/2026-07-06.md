## Findings (ranked by impact)

1. **The app claims reliability it hasn't verified — against its own core promise.**
   Detail screen shows *"Reliable alarms are on. Takna will fire even in battery-saver mode."* unconditionally (`reminder_detail_screen.dart:186`), and the home hero says *"Alarm set · notifies even when your phone is locked"* (`home_screen.dart:245`) — even when notification or exact-alarm permission is denied (Android 13+ denies notifications by default; Android auto-revokes permissions from unused apps). A user who skipped or lost permissions is told everything is fine, then the alarm silently never rings. **Fix: plan-worthy (small)** — make those strings permission-aware and show a home warning when notifications/exact alarms are missing. *(Challenged below.)*

2. **Editing a reminder silently re-enables it and resets `createdAt`.**
   `_save()` always writes `isEnabled: true` and `createdAt: now` (`add_edit_reminder_screen.dart:112,114`). Edit a paused reminder → it turns back on without the user asking. **Fix: tiny** — preserve both fields from the loaded reminder on edit.

3. **Fired one-time reminders live in the list forever as "No upcoming".**
   Nothing ever disables or clears a one-time reminder after it fires; the list accumulates dead rows. **Fix: plan-worthy (small)** — see feature suggestion.

4. **Alarm dismiss never reconciles.**
   Architecture doc says re-arm on alarm fire. `_dismiss()` (`alarm_screen.dart:50`) and the background dismiss action (`notification_service.dart:63` ignores non-snooze actions) don't call `reconcile()`. The 10-per-reminder window and app-open reconcile mask this, but a heavy-snooze user who rarely opens the app can drain the window. **Fix: tiny** — reconcile on dismiss in the alarm screen; optionally handle dismiss in the background handler.

5. **Settings permission rows don't refresh on app resume.**
   `_load()` runs on init and after taps (`settings_screen.dart:29`), but granting a permission in system settings and returning shows stale "Allow" buttons. **Fix: tiny** — reload on `AppLifecycleState.resumed`.

6. **Scheduler and payload logic are untested.**
   Only recurrence has tests. `Scheduler.reconcile()` window/snooze/offset logic, `occurrenceNotificationId` stability, and `parsePayload` (title-with-`|` round-trip) are the reliability-critical paths. **Fix: tiny-to-small** — one test file with a fake `NotificationService` and in-memory DB (`AppDatabase.forTesting` already exists).

7. **Inconsistent snooze default:** DB default 10 (`database.dart:14`), UI and payload fallback 5. Harmless today (UI always writes a value) but a trap. **Fix: tiny** — align on one value.

8. **Raw exceptions shown to users:** `error: (e, _) => Center(child: Text('$e'))` on home and detail. **Fix: tiny** — friendly message.

## Challenged: finding #1 (honest reliability status)

- **Assumptions tested:** (a) permission state is queryable — yes, settings already does it via `permission_handler`; (b) users actually end up permission-less — real on Android 13+ (default-deny notifications, auto-revoke on unused apps); (c) a banner changes behavior — unproven but the cost is one card.
- **What got weaker:** a *battery-optimization* warning is unreliable (`ignoreBatteryOptimizations.isGranted` misreports on some OEMs) and risks nag fatigue — drop it from the banner, keep it in settings only. Also, a "home banner subsystem" is over-scoped: the cheapest honest version is (1) make the two hardcoded strings conditional, (2) one dismissible home card shown only when notifications or exact alarms are denied — the two hard failures.
- **What survived:** the core — an alarm app must not display "reliable mode on" text it never checked. Verdict: **Pursue (narrowed)**. Validation: deny notifications on an Android 13 emulator, create a reminder, observe the app currently claiming reliability while the alarm never fires.

## Feature suggestion (`--feature`)

**Auto-complete one-time reminders after they fire** (Google Clock pattern: a fired one-time alarm flips its toggle off). Grounded in finding #3 — the only unbounded-clutter path in the app today.

- **Challenged:** auto-*hiding* or archiving lost — a user who slept through the alarm must still see the row (missed ≠ done), and an archive/history screen is YAGNI. Handling it in dismiss paths lost too — dismiss-from-shade runs (or doesn't run) a background isolate, giving three inconsistent paths. What survived is the lazy root-cause version: **`reconcile()` disables one-time reminders whose fire time has passed** — one write in the function every path already routes through; the row stays visible with its toggle off, and re-enabling is meaningless-but-harmless. Risk: a reminder that fired while the phone was off gets marked past on next reconcile — acceptable, the time *has* passed.
- **Verdict: Pursue (small)** — a few lines in `Scheduler.reconcile()` + one test.
